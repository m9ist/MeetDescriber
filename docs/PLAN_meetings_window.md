# ТЗ: Окно управления совещаниями

> **Статус: реализовано** — коммиты `c332a34` (фаза 1), `9ad4c34` (фаза 2),
> `d2444a3` (фаза 3). Acceptance-критерии в конце документа отмечены.

## Цель

Дать пользователю единое окно для просмотра и управления всеми
совещаниями: открыть документы, перезапустить этапы, удалить
запись/совещание, очистить старое аудио.

## Точка входа

Новый пункт в меню трея — между «Обработанные» и «Начать запись вручную»:

```
Ожидание
──────────────
Необработанные (10) ▶
Обработанные (16)   ▶
Все совещания…       ← новый пункт
──────────────
Начать запись вручную
Выход
```

Клик открывает окно `MeetingsWindow`. Окно немодальное.

## UI

Окно tkinter `Toplevel` с:
- заголовок: «for_meets — Все совещания»
- размер: ~900×600 (с прокруткой)
- закрытие по крестику работает как обычно

### Структура окна (сверху вниз)

```
┌─ Поиск и фильтр ─────────────────────────────────┐
│ [🔍 Поиск по названию____]  Статус: [Все ▼]      │
├─ Таблица (Treeview со скроллом) ─────────────────┤
│ Дата ↓        Название           Длит.   Статус  │
│ 2026-05-06    Avis ИИзация       42:11   done    │
│ 2026-05-06    Данные стриминга   1:23:04 transcribed│
│ ...                                               │
├─ Низ: статистика + глобальные действия ──────────┤
│ Всего: 73 совещания · 12.4 GB записей · 280 MB   │
│ документов        [Удалить аудио старше 14 дней] │
└──────────────────────────────────────────────────┘
```

### Колонки таблицы

| Колонка     | Источник                                          |
|-------------|---------------------------------------------------|
| Дата        | `sessions.started_at` (формат `YYYY-MM-DD HH:MM`) |
| Название    | `sessions.title`                                  |
| Длительность| `sessions.ended_at − started_at`, формат `MM:SS` или `H:MM:SS` |
| Статус      | `jobs.status` (на русском: ожидает / обработка / транскрибировано / проанализировано / готово / ошибка) |

Сортировка: только по умолчанию — по дате убывание (новые сверху).
Кликабельные заголовки **не делаем**.

## Поиск и фильтр

### Поиск
Текстовое поле сверху. Фильтр по подстроке в названии (case-insensitive).
Реакция мгновенная (на каждое нажатие клавиши, debounce не нужен).

### Фильтр по статусу
Dropdown справа от поиска, варианты:
- Все
- Ожидает (pending)
- Транскрибировано (transcribed)
- Проанализировано (analyzed)
- Готово (done)
- Ошибка (error)

Поиск и фильтр комбинируются (AND).

## Действия (контекстное меню по правой кнопке)

```
Открыть транскрипцию
Открыть анализ          ← disabled если файла нет
Открыть follow-up       ← disabled если файла нет
──────────────
Редактировать название и агенду
──────────────
Перезапустить транскрипцию  ← удаляет transcription/analysis/followup, status=pending
Перезапустить анализ        ← удаляет analysis/followup, status=transcribed
Перезапустить follow-up     ← удаляет followup, status=analyzed
──────────────
Удалить аудио               ← disabled если status < transcribed (потеряем данные)
Удалить совещание полностью ← через подтверждение
```

### Семантика действий

**«Перезапустить N»** = удалить файлы этого и последующих этапов +
выставить соответствующий status. Не запускает обработку
автоматически — после перезапуска совещание появится в трее как
необработанное, оттуда же запускается.

**«Удалить аудио»**:
- Доступно только если `status IN ('transcribed', 'analyzed', 'done')`
  (транскрипция уже есть, аудио уже не нужно)
- Удаляет папку `data/recordings/session_X/` целиком
- Документы остаются
- Подтверждение через `messagebox.askyesno`

**«Удалить совещание полностью»**:
- Удаляет всё: запись в `sessions`, в `jobs`, в `speakers`,
  все документы (`*_transcription.md`, `*_analysis.md`,
  `*_followup.md`, `*_*_prompt.md`), папку `session_X/`
- Подтверждение через `messagebox.askyesno` с явным указанием
  что будет удалено: «Удалить совещание "X"? Будут стёрты:
  3 документа, 12 аудио-чанков (2.1 GB)»

## Глобальные действия (внизу окна)

### Статистика

Строка слева, обновляется при каждом открытии окна:

```
Всего: 73 совещания · 12.4 GB записей · 280 MB документов
```

Источники:
- Совещаний — `SELECT COUNT(*) FROM sessions`
- GB записей — сумма размеров всех `data/recordings/session_*` папок
- MB документов — сумма размеров всех файлов в `data/documents/`

### Кнопка «Удалить аудио старше 14 дней»

Справа внизу. По клику:
1. Найти все session_X где `started_at < now - 14 days` И
   `status IN ('transcribed', 'analyzed', 'done')`
2. Показать подтверждение: «Найдено 12 папок аудио (3.8 GB).
   Удалить?»
3. Удалить папки `session_X/`

Документы при этом не трогаем.

## Скрытие совещаний из «Необработанных»

Иногда хочется убрать совещание из активной очереди обработки, не
завершая его до конца (например, транскрипция есть, но анализ
делать не надо). Совещание перестаёт мозолить глаза в трее, при этом
остаётся в системе и доступно из окна управления.

### Поведение

- В трее в подменю «Необработанные» каждой строки появляется пункт
  **«Скрыть из списка»**
- По клику: `dismissed=1`, `updated_at=now()` → совещание исчезает
  из «Необработанных» и появляется в верху «Обработанных»
- В «Обработанных» показывается ровно как обычные `done`:
  по `updated_at DESC LIMIT 20`. Когда уезжает за лимит — пропадает
  из трея, но остаётся в БД и в окне «Все совещания»
- Сброс статуса из окна «Все совещания» (см. ниже) автоматически
  снимает флаг `dismissed`

### Изменения SQL-запросов трея

```sql
-- Необработанные: было
WHERE status IN ('pending', 'transcribed', 'analyzed')

-- Стало
WHERE status IN ('pending', 'transcribed', 'analyzed') AND dismissed = 0

-- Обработанные: было
WHERE status = 'done'

-- Стало
WHERE status = 'done' OR dismissed = 1
```

### `_job_label` для скрытых

Скрытое совещание попадает в «Обработанные», но `status` у него
не `done`. Текущий `_job_label` для `transcribed`/`analyzed`
добавляет суффикс `→ анализ`/`→ follow-up` — в контексте
«Обработанных» это вводит в заблуждение. Решение: `_job_label`
получает аргумент `context: Literal["pending", "done"]` и не
добавляет суффикс при `context="done"`.

### Действие сброса (в окне «Все совещания»)

Контекстное меню `Перезапустить транскрипцию/анализ/follow-up` и
так возвращает совещание к нужному этапу. Дополнительно нужно:
**`UPDATE jobs SET dismissed=0 WHERE id=…`** при любом перезапуске —
чтобы пользователь снова увидел его в «Необработанных».

## Изменения в схеме БД

### Новые колонки в `jobs`

Для отображения длительностей этапов в hover-попапе:

```sql
ALTER TABLE jobs ADD COLUMN transcribe_duration_sec REAL;
ALTER TABLE jobs ADD COLUMN diarize_duration_sec REAL;
ALTER TABLE jobs ADD COLUMN analyze_duration_sec REAL;
ALTER TABLE jobs ADD COLUMN followup_duration_sec REAL;
```

Заполнять в `pipeline.py` — там уже измеряем `time.monotonic()`,
просто пишем в БД после каждого этапа.

Для скрытия из «Необработанных»:

```sql
ALTER TABLE jobs ADD COLUMN dismissed INTEGER DEFAULT 0;
```

### Новая колонка в `sessions`

```sql
ALTER TABLE sessions ADD COLUMN ended_at TEXT;
```

Заполнять в `_on_stop_manual` / `_on_stop_extension` в `main.py`
при остановке записи.

Для уже существующих сессий — оставляем NULL, в окне показываем
длительность как «—» если NULL.

Миграция: добавить блок в `init_db()` с проверкой через
`PRAGMA table_info(jobs)` (паттерн уже использовался ранее).

## Реализация: этапы

1. ✅ **Миграция БД** — `app/storage/db.py`: добавлен блок
   `PRAGMA table_info` + ALTER TABLE для 5 колонок в `jobs`
   (длительности, `dismissed`) и `ended_at` в `sessions`.
   Идемпотентно. Коммит `c332a34`.

2. ✅ **Сохранение длительностей** — `app/processing/pipeline.py`:
   после каждого реально выполненного этапа делается
   `UPDATE jobs SET *_duration_sec=…`. Cache hit не перезаписывает.
   Коммит `c332a34`.

3. ✅ **Сохранение `ended_at`** — оказалось уже было реализовано в
   `_stop_and_offer_processing`, миграция добавлена для старых БД.

4. ✅ **Скрытие из «Необработанных»** — коммит `9ad4c34`:
   - `app/main.py`: SQL-запросы трея учитывают `dismissed`
   - `app/ui/tray.py`: `_job_label(job, context)`, в pending
     submenu добавлен пункт «Скрыть из списка»
   - коллбэк `on_dismiss_job(job_id)` в `App` → `set_dismissed`
     + `_refresh_tray_jobs()`

5. ✅ **Бэкенд-функции** — новый `app/storage/meetings_repo.py`,
   все функции из ТЗ + `count_old_audio` для предпросмотра кнопки
   удаления старого аудио. Коммит `c332a34` + `d2444a3`.

6. ✅ **UI** — `app/ui/meetings_window.py` (~550 строк), коммит
   `d2444a3`:
   - `MeetingsWindow(parent, on_data_changed=None)`
   - `Treeview` 4 колонки + scrollbar
   - поиск (мгновенный, через `trace_add`) + фильтр статуса
   - правый клик → динамическое контекстное меню (12 пунктов,
     `state="disabled"` пересчитывается при каждом клике)
   - hover-tooltip с задержкой 500 мс через `after`/`after_cancel`
   - статистика + кнопка «Удалить аудио старше 14 дней»
   - `on_data_changed` — коллбэк для обновления трея после
     destructive-действий

7. ✅ **Точка входа** — `app/ui/tray.py`: пункт меню «Все совещания…»
   + коллбэк `on_open_meetings_window`. Коммит `d2444a3`.

8. ✅ **Подключение в main.py** — передан `on_open_meetings_window`
   и `on_data_changed=self._refresh_tray_jobs` в `MeetingsWindow`.
